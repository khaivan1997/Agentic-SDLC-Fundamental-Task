1.1  Make sure Docker Desktop is running on your Mac.

1.2  Open a terminal and run:

       cd /Users/khai.a.ngo/Projects/Training/agent-garage
       docker compose --profile cpu up

     NOTE: You're on macOS, so GPU passthrough is not available.
     Use "--profile cpu" to run Ollama inside Docker on CPU.

     ALTERNATIVELY (faster inference): Install Ollama natively on your Mac

────────────────────────────────────────────────────────────────────────────────
  USE CASE A: Pull Request Description Generator
────────────────────────────────────────────────────────────────────────────────

  What it does:
    You send commit messages → LLM returns a clean PR description.

  4A.1  In N8N, click "+ Add Workflow" → name it "PR Description Generator".

  4A.2  Add a Webhook node:
        - HTTP Method: POST
        - Path: pr-description
        - Response Mode: "Using 'Respond to Webhook' node"

  4A.3  Add an HTTP Request node (the LLM API call):
        - Method: POST
        - URL: http://ollama:11434/api/generate
          (if native Ollama: http://host.docker.internal:11434/api/generate)
        - Body Content Type: JSON
        - Body:
          {
            "model": "llama3.2",
            "prompt": "You are a senior software engineer.\nGenerate a professional Pull Request description based on these commit messages:\n\n{{ $json.body.commits }}\n\nInclude: a PR title, a summary paragraph, and a bullet list of changes.",
            "stream": false
          }

  4A.4  Add a "Respond to Webhook" node:
        - Response Body: {{ $json.response }}
          (the field name depends on Ollama's output – check the actual key)

  4A.5  Connect: Webhook → HTTP Request → Respond to Webhook

  4A.6  Save and activate the workflow (toggle "Active").

  4A.7  TEST – open a terminal and run:

        curl http://localhost:5678/webhook/pr-description \
          --request POST \
          --header "Content-Type: application/json" \
          --data '{
            "commits": "feat: add task CRUD endpoints\nfix: resolve CORS issue\nrefactor: extract TaskService\ntest: add integration tests\ndocs: update README"
          }'

  4A.8  You should receive a formatted PR description from the LLM.

────────────────────────────────────────────────────────────────────────────────
  USE CASE B: Code Review Suggestions
────────────────────────────────────────────────────────────────────────────────

  What it does:
    You send a code snippet or diff → LLM returns review comments.

  4B.1  In N8N, click "+ Add Workflow" → name it "Code Review Bot".

  4B.2  Add a Webhook node:
        - HTTP Method: POST
        - Path: code-review
        - Response Mode: "Using 'Respond to Webhook' node"

  4B.3  Add an HTTP Request node:
        - Method: POST
        - URL: http://ollama:11434/api/generate
        - Body (JSON):
          {
            "model": "llama3.2",
            "prompt": "You are an experienced code reviewer.\nReview the following code and provide:\n1. Overall quality assessment\n2. Potential bugs or issues\n3. Suggestions for improvement\n4. Missing test coverage\n\nCode:\n{{ $json.body.code }}",
            "stream": false
          }

  4B.4  Add a "Respond to Webhook" node mapping the LLM response.

  4B.5  Connect: Webhook → HTTP Request → Respond to Webhook

  4B.6  Save and activate.

  4B.7  TEST:

        curl http://localhost:5678/webhook/code-review \
          --request POST \
          --header "Content-Type: application/json" \
          --data '{
            "code": "public class TaskService {\n  @Autowired\n  TaskRepository repo;\n\n  public Task create(Task t) {\n    if (t.getTitle() == null) throw new RuntimeException(\"no title\");\n    return repo.save(t);\n  }\n}"
          }'

────────────────────────────────────────────────────────────────────────────────
  USE CASE C: User Story → Acceptance Criteria (Gherkin)
────────────────────────────────────────────────────────────────────────────────

  What it does:
    You send a user story → LLM returns Gherkin acceptance criteria.

  4C.1  In N8N, click "+ Add Workflow" → name it "Gherkin Generator".

  4C.2  Add a Webhook node:
        - HTTP Method: POST
        - Path: gherkin
        - Response Mode: "Using 'Respond to Webhook' node"

  4C.3  Add an HTTP Request node:
        - Method: POST
        - URL: http://ollama:11434/api/generate
        - Body (JSON):
          {
            "model": "llama3.2",
            "prompt": "You are an Agile Product Owner.\nConvert the following user story into Gherkin-format acceptance criteria.\nUse Given/When/Then syntax.\nProvide at least 3 scenarios including a happy path and edge cases.\n\nUser Story:\n{{ $json.body.userStory }}",
            "stream": false
          }

  4C.4  Add a "Respond to Webhook" node mapping the LLM response.

  4C.5  Connect: Webhook → HTTP Request → Respond to Webhook

  4C.6  Save and activate.

  4C.7  TEST:

        curl http://localhost:5678/webhook/gherkin \
          --request POST \
          --header "Content-Type: application/json" \
          --data '{
            "userStory": "As a user, I want to change my password so I can secure my account."
          }'

  4C.8  Expected output (example):

        Feature: Change Password
          Scenario: Successful password change
            Given the user is logged in
            When the user navigates to account settings and submits a new password
            Then the password should be updated and a confirmation message displayed

          Scenario: Password too short
            Given the user is logged in
            When the user submits a password shorter than 8 characters
            Then an error message "Password must be at least 8 characters" is shown

================================================================================
  TROUBLESHOOTING
================================================================================

  N8N can't connect to Ollama:
    → Native Ollama on Mac: credential URL must be
      http://host.docker.internal:11434/
    → Docker Ollama (--profile cpu): URL must be http://ollama:11434/

  LLM is very slow (30-60+ seconds):
    → llama3.2 on CPU is slow. This is normal. Be patient.
    → For faster results, use a smaller model like "phi3" or "tinyllama".

  Model not found / 404 error:
    → The model may still be downloading. Check with:
      docker exec -it ollama ollama list
    → Or pull manually: docker exec -it ollama ollama pull llama3.2

  OpenWebUI blank page:
    → Refresh the page. Sometimes it has display delays on startup.

  Containers won't start:
    → Ensure Docker Desktop is running.
    → Check port conflicts: lsof -i :5678  and  lsof -i :3000
    → Reset: docker compose down && docker compose --profile cpu up
